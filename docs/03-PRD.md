# Nove AI 小说创作平台：产品需求文档

> 文档状态：Draft 1.0  
> 更新日期：2026-07-15  
> 目标版本：MVP  
> 关联文档：[技术栈](./01-TECH-STACK.md) · [总体设计](./02-SYSTEM-DESIGN.md) · [UI 设计](./04-UI-DESIGN.md)

## 1. 产品概述

Nove 是一个面向长篇小说作者的 AI 创作工作台。它不仅生成文字，还持续维护小说的大纲、人物、地点、世界规则、事件、时间线、伏笔和人物知识边界，并在每次生成后使用可配置的审计 AI 打分。

产品核心承诺：

> AI 可以参与写作，但小说事实、人物命运和最终版本始终由作者控制。

## 2. 问题背景

普通 AI 对话工具用于写小说时存在以下问题：

- 小说变长后遗忘早期章节。
- 人物性格、能力、伤势、位置和说话方式不稳定。
- 角色提前知道不该知道的信息。
- 地点和世界规则前后矛盾。
- 生成内容偏离大纲，章节没有实际推进。
- 转折依赖临时新增设定，缺少前文铺垫。
- 语言出现模板化排比、总结句和同质化 AI 表达。
- 修改旧章节后，无法判断后续哪些章节受到影响。
- 作者无法稳定比较不同模型、提示词和重写版本。

## 3. 产品目标

### 3.1 用户目标

- 作者可以从创意、大纲或已有正文开始创建小说。
- 每章生成前，系统自动找到相关人物、地点、事件和历史内容。
- 作者能够明确控制本章必须发生、不能发生的内容。
- 系统能够发现连续性、人物一致性和大纲偏离问题。
- 审计低分时可以自动局部修改或整章重写。
- 作者可以查看每次修改原因、评分和版本差异。
- 已确认章节自动沉淀为后续写作可检索的长期记忆。

### 3.2 业务目标

MVP 阶段不以收入为首要目标，优先验证：

- 用户是否愿意在同一项目中持续写作 10 章以上。
- 用户是否认可故事圣经和连续性检查带来的价值。
- AI 审计是否能降低废稿率。
- 用户是否愿意确认 AI 生成版本而非复制到外部编辑器。

### 3.3 成功指标

| 指标 | MVP 目标 |
|---|---:|
| 建立项目后完成首章生成的比例 | 不低于 60% |
| 连续完成 10 章的活跃项目比例 | 不低于 25% |
| AI 版本被接受或局部修改后接受的比例 | 不低于 50% |
| 首次审计通过率 | 记录基线，不作为越高越好 |
| 三轮内审计通过率 | 不低于 85% |
| 用户手动报告的严重连续性错误 | 每 10 章少于 1 个 |
| 自动保存导致的数据丢失 | 0 |

## 4. 非目标

MVP 不包含：

- 小说阅读社区和公开发布平台。
- 多人实时协作。
- 自动代替作者连续生成整本小说并直接发布。
- 版权检测、抄袭检测和商业出版合规保证。
- 完整的排版出版系统。
- 语音输入、配音和视频生成。
- 移动端完整写作体验。

## 5. 目标用户

### 5.1 网文作者

- 高频写作。
- 关注章节节奏、钩子和稳定产量。
- 希望大纲约束清晰，自动检查断点和爽点。

### 5.2 长篇类型小说作者

- 重视人物成长、世界观和伏笔。
- 需要跨卷记忆、时间线和关系变化。
- 更常使用局部改写而非整章自动生成。

### 5.3 AI 辅助写作用户

- 有创意但缺少完整规划经验。
- 需要人物构建、大纲展开和章节建议。
- 希望获得明确评分和修改理由。

## 6. 核心产品概念

| 概念 | 定义 |
|---|---|
| 小说项目 | 一部小说的全部数据边界 |
| 故事圣经 | 人物、地点、势力、物品、规则和术语 |
| 大纲树 | 卷、剧情弧、章节、场景组成的层级结构 |
| 章节任务 | 当前章节目标、冲突、必达事件、禁区和结尾钩子 |
| 权威事实 | 已由用户确认或锁定的结构化事实 |
| 长期记忆 | 已确认正文及其摘要、事件和向量片段 |
| 剧情线 | 伏笔、谜团、承诺、冲突和关系发展 |
| 审计 | 独立 AI 按评分表检查正文并提供证据 |
| 致命问题 | 无论总分多少都必须处理的事实或规则错误 |
| Skill | 具有输入输出约束、权限和版本的可插拔能力 |

## 7. 核心用户流程

### 7.1 从零开始

1. 新建小说。
2. 填写题材、主题、核心创意、目标字数和文风。
3. AI 生成故事方向和总纲草案。
4. 用户编辑并确认卷、剧情弧和章节大纲。
5. 创建或生成主要人物、地点和世界规则。
6. 打开第一章，补充章节目标。
7. AI 生成正文。
8. 审计 AI 评分。
9. 系统局部修改或整章重写，直到通过或达到次数上限。
10. 用户编辑并确认章节。
11. 系统提取事实、状态、事件、伏笔并写入记忆。

### 7.2 从已有小说继续

1. 新建项目并选择导入。
2. 按章节标题或分隔符切分正文。
3. AI 提取人物、地点、大纲、事件和摘要。
4. 用户在导入审阅页纠正实体合并和章节顺序。
5. 确认后建立故事圣经与向量索引。
6. 从下一章继续写作。

### 7.3 修改旧章节

1. 用户编辑已确认章节。
2. 系统创建新版本并提示该章会影响正式记忆。
3. 用户确认重新提交。
4. 系统计算事实差异和受影响章节。
5. 后续章节显示待检查标记。
6. 用户逐章重新审计、局部修改或忽略影响。

## 8. 功能优先级

- P0：MVP 必须完成。
- P1：MVP 后的首个增强版本。
- P2：长期规划。

## 9. 功能需求

### FR-001 小说项目管理

优先级：P0

需求：

- 创建、重命名、归档、导出和删除小说项目。
- 配置题材、主题、核心创意、目标字数、叙事视角、时态和语言。
- 配置写作硬规则和禁止内容。
- 首页展示最近编辑时间、章节进度、总字数和待处理审计数。

验收标准：

- 新项目必须在 3 个步骤内进入工作台。
- 未填写的高级配置不阻塞创建。
- 删除项目需要二次确认，并提供短时撤销入口。
- 项目之间的数据和向量检索完全隔离。

### FR-002 大纲管理

优先级：P0

需求：

- 支持卷、剧情弧、章节和场景四级节点。
- 支持拖动排序、折叠、复制和批量生成章节。
- 每章保存目标、冲突、必达事件、禁止事件、人物、地点、亮点、转折和钩子。
- AI 可以根据当前总纲生成下一级节点。
- 用户可以锁定节点，防止 AI 重写。

验收标准：

- 大纲节点调整顺序后，章节编号与依赖关系可重新计算。
- 锁定内容不会被后续 AI 规划覆盖。
- 没有章节目标时，系统可以生成，但必须提示大纲约束较弱。

### FR-003 人物管理

优先级：P0

需求：

- 创建人物画像：姓名、别名、年龄、外貌、背景、性格、动机、恐惧、秘密、能力、弱点、说话方式。
- 展示人物当前状态：位置、身体、情绪、已知信息、信念、物品和关系。
- AI 可以生成人物草案，但用户确认后才成为权威设定。
- 支持人物字段锁定。

验收标准：

- 人物历史状态可以按章节查看。
- 修改固定画像时提示可能影响的章节。
- 同名人物或别名冲突时要求用户合并或区分。

### FR-004 地点与世界设定

优先级：P0

需求：

- 管理地点、区域层级、氛围、地理关系、规则和可用时间。
- 管理势力、物品、能力体系、历史和术语。
- 大纲选择地点后，生成上下文自动加入相关设定。
- 支持地点毁坏、封锁、占领等章节状态。

验收标准：

- 已销毁地点不能在后续章节中无解释恢复。
- 地点层级和人物位置可以被连续性检查读取。
- 用户锁定的世界规则不可由 AI 自动修改。

### FR-005 章节编辑器

优先级：P0

需求：

- 支持正文编辑、自动保存、字数统计、查找替换和全屏专注模式。
- 支持选区续写、扩写、缩写、改写、对话优化和文风优化。
- AI 修改以差异形式展示，用户可以接受、拒绝或逐段处理。
- 支持锁定段落，整章重写时必须保留。

验收标准：

- 输入期间不发生光标跳动。
- 自动保存失败时显示明确状态并保留本地草稿。
- AI 候选内容不会直接覆盖用户正文。
- 章节恢复历史版本时先保存当前版本。

### FR-006 章节生成

优先级：P0

需求：

- 根据章节大纲、故事圣经、最近章节和检索记忆生成正文。
- 用户可设置目标字数、叙述节奏、对话比例和文风。
- 生成过程显示阶段并支持取消。
- 生成完成后自动进入审计，除非用户关闭自动审计。

验收标准：

- 生成上下文不得包含后续章节正文。
- 取消任务后不能覆盖编辑器当前内容。
- 生成稿必须保存为独立版本。
- 本章必达事件和禁止事件随请求传入。

### FR-007 长期记忆与向量知识库

优先级：P0

需求：

- 已确认章节自动切分、摘要、向量化。
- 生成前自动检索相关人物、地点、事件、伏笔和场景。
- 提供知识库同步状态和重建索引入口。
- 允许用户查看本次生成实际引用的记忆来源。

验收标准：

- 未确认草稿不能进入正式向量库。
- 检索必须以小说项目隔离。
- 向量索引失败不丢失章节正文，并可以重试。
- 更换 Embedding 模型时提示重建索引。

### FR-008 连续性检查

优先级：P0

需求：

- 检查人物位置、生死、伤势、知识边界、物品归属、时间线和世界规则。
- 检查人物语言和行为是否明显偏离画像。
- 每个问题提供类型、严重级别、正文证据、依据和建议。
- 用户可以接受修改、忽略一次或将其标记为有意设定。

验收标准：

- 硬事实冲突优先于文风问题显示。
- 用户标记为有意设定后，系统要求补充解释或更新权威事实。
- 问题定位能够跳转到对应正文位置。

### FR-009 AI 审计评分

优先级：P0

需求：

- 用户可以选择独立审计模型。
- 可以配置评分维度、权重、通过分数、局部修改分数和最大重写次数。
- 默认评分维度为连续性、人物一致性、大纲完成度、剧情推进、冲突张力、亮点转折、文笔和 AI 痕迹。
- 审计结果保存总分、维度分、证据、问题、优点和重写要求。
- 审计模型温度默认 0.1。

验收标准：

- 所有扣分项必须引用正文证据。
- 分数达到通过阈值且无致命问题才算通过。
- 评分表变化后生成新的 rubric_version，不覆盖历史审计。
- 写作模型与审计模型可以使用不同供应商。

### FR-010 低分自动修改和重写

优先级：P0

需求：

- 中等分数执行局部修改。
- 低分或致命问题执行整章重写。
- 重写必须接收 must_preserve、must_improve、must_not_include 和 locked_ranges。
- 每轮生成独立版本并重新审计。
- 达到最大次数后进入人工审阅。

验收标准：

- 自动循环默认最多 3 次。
- 用户可以随时停止循环。
- 锁定内容被破坏时，该版本直接失败。
- UI 可以对比每轮分数和正文差异。

### FR-011 亮点、转折和伏笔

优先级：P0

需求：

- 每章大纲可以设置亮点、转折、揭示和结尾钩子。
- 转折必须保存表面预期、真实变化和支撑线索。
- 伏笔保存种下章节、目标回收章节、线索、状态和重要度。
- 系统提示长期未发展的高重要度伏笔。

验收标准：

- 审计能够判断本章是否完成要求的亮点或转折。
- 转折没有任何前文支撑时需要扣分并提示。
- 回收伏笔后更新状态，但保留历史线索。

### FR-012 Skill 管理

优先级：P0

需求：

- 展示已安装 Skill、版本、描述、权限和启用状态。
- 为每个 Agent 配置允许调用的 Skill。
- 支持手动调用和工作流自动调用。
- 记录输入摘要、输出摘要、耗时、状态和错误。

验收标准：

- 未授权 Agent 无法调用 Skill。
- 输入输出不符合 Schema 时调用失败且不写业务数据。
- Skill 超时不会导致章节正文丢失。
- 用户可以针对单本小说关闭某个 Skill。

### FR-013 模型配置

优先级：P0

需求：

- 配置供应商、模型名、Base URL、API Key、温度和最大输出。
- 测试连接。
- 分别选择大纲、写作、审计、连续性、润色、提取和 Embedding 模型。
- 展示调用 Token、估算费用、延迟和错误率。

验收标准：

- API Key 在 UI 中只显示掩码。
- 测试连接失败时提供可恢复的错误信息。
- 删除正在使用的模型配置前必须先替换绑定。

### FR-014 版本和恢复

优先级：P0

需求：

- 保存 AI 生成、AI 修改、AI 重写、用户手改和确认版本。
- 展示版本来源、时间、模型、审计分数和字数。
- 支持双栏 diff 和恢复。

验收标准：

- 恢复前自动保存当前版本。
- 历史审计始终指向当时的正文版本。
- 删除普通版本不能删除当前确认版本。

### FR-015 导入与导出

优先级：P1

需求：

- 导入 TXT、Markdown 和 DOCX。
- 识别章节标题并允许手动调整切分。
- 导出整书 Markdown、TXT 和 DOCX。
- 可选导出故事圣经、大纲和审计报告。

### FR-016 时间线和人物关系图

优先级：P1

需求：

- 以章节或故事时间查看事件。
- 过滤人物、地点和剧情线。
- 展示人物关系及每次变化的章节依据。

### FR-017 全书审计

优先级：P1

需求：

- 按卷或整书扫描连续性、重复情节、节奏、人物消失和伏笔遗漏。
- 输出问题清单，不自动批量改写整书。

### FR-018 多人协作

优先级：P2

- 角色权限、评论、变更提议和审批。
- 不在 MVP 数据模型中依赖单一用户字段，使用 workspace 边界预留扩展。

## 10. 审计默认配置

    enabled: true
    pass_score: 85
    revise_score: 70
    max_rewrite_attempts: 3
    auto_audit: true
    auto_revise: true
    auto_rewrite: true
    fatal_issue_force_rewrite: true
    audit_temperature: 0.1

用户可以关闭自动重写，但不能关闭权威规则检查；关闭后问题只显示，不自动修改。

## 11. 状态与权限规则

- AI 草稿不能自动成为已确认章节。
- 已确认章节修改后必须重新提交记忆。
- 用户手写和锁定内容优先级最高。
- 审计通过不等于用户确认。
- 忽略致命问题时，必须记录用户理由或更新权威事实。
- 任何自动修改都必须可以撤销。

## 12. 非功能需求

### 性能

- 工作台首屏可交互时间在常规网络下不超过 3 秒。
- 编辑器输入响应目标低于 50 毫秒。
- 自动保存静默执行，不阻塞输入。
- 1000 个章节节点下大纲树仍可流畅滚动。
- 生成超过 1 秒时显示阶段进度。

### 可靠性

- 正文自动保存失败时保留本地恢复副本。
- 长任务可取消且不会覆盖新版本。
- 数据库和向量索引状态可独立恢复。
- 每次 AI 修改均形成版本。

### 安全与隐私

- 模型密钥加密保存。
- 用户可选择仅使用本地模型。
- 日志默认不记录完整正文。
- 项目导出和删除均需要明确操作。

### 无障碍

- 键盘可完成主要工作流。
- 正常文本对比度至少 4.5:1。
- 图标按钮有可访问名称和 Tooltip。
- 状态不能只用颜色表达。
- 支持减少动态效果。

## 13. 数据埋点

仅收集产品行为和性能，不默认上传小说正文：

- project_created
- outline_generated
- chapter_generated
- audit_completed
- revision_started
- rewrite_started
- version_accepted
- chapter_confirmed
- memory_indexed
- continuity_issue_ignored
- job_cancelled

埋点包含项目匿名 ID、功能、耗时、模型类型和结果状态，不包含正文内容。

## 14. MVP 发布验收

发布前必须通过：

1. 创建小说并生成大纲。
2. 创建人物、地点和世界规则。
3. 连续生成并确认至少 10 章。
4. 第 10 章能够检索第 1 章的重要事实。
5. 人为制造人物知识泄漏后能被检查发现。
6. 审计低于阈值后能够自动重写并再次评分。
7. 三轮未通过后正确停止。
8. 用户编辑期间旧任务不能覆盖新内容。
9. 修改旧章节后能标记受影响的后续章节。
10. 可以恢复任一保留版本。
11. Skill 权限和 Schema 校验有效。
12. 关闭云端模型后可以使用本地模型完成核心流程。

## 15. 风险与应对

| 风险 | 应对 |
|---|---|
| 审计模型评分波动 | 低温度、固定评分表版本、要求证据、可选重复审计 |
| 自动重写越改越偏 | 锁定事实、保护段落、最大轮次、每轮重新检查 |
| 向量召回遗漏关键事实 | 结构化事实优先、实体过滤、最近章节保底 |
| 成本过高 | 上下文预算、模型角色分层、缓存相同版本审计 |
| 修改旧章引发连锁变化 | 影响分析和待检查状态，不自动覆盖后文 |
| 用户过度依赖 AI | 所有正式提交由用户确认，保留来源和版本 |
| 文风同质化 | 作者文风档案、局部示例、独立润色与 AI 痕迹评分 |
